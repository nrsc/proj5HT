# 5-HT Puff Response Classifier — Improvement Plan

> Generated by auditing every classifier iteration in `dev/classify_pct_change/`,
> `dev/build_response_classifier/`, and the promoted `R/classify5HT*.R` files.

---

## 1. Executive summary

The promoted classifier (`R/classify5HT.R`) was distilled from
`dev/build_response_classifier/` — a cleaner but **less feature-rich** rewrite.
Several capabilities present in the most mature iteration
(`dev/classify_pct_change/df_prep_new.R`) were dropped during promotion.
The plotting layer (`R/plot_classify5HT.R`) was likewise simplified and lost
subtype faceting, proper trace colouring, and the spline / smooth-mean overlay
that made trace figures interpretable.

This document catalogues **what is broken or missing**, **why it matters**, and
**concrete steps to fix each issue**, ordered so the highest-impact items come
first.

---

## 2. Classification logic — features lost during promotion

### 2.1 Early / late window split (slow-biphasic detection)

| Feature | `df_prep_new.R` | Promoted `R/classify5HT.R` |
|---|---|---|
| Early window (0–12 s) | ✔ `early_window = c(0, 12)` | ✘ absent |
| Late window (12–56 s) | ✔ `late_window = c(12, 56)` | ✘ absent |
| Per-window persistence flags | ✔ `early_has_exc`, `late_has_inh`, etc. | ✘ absent |
| `slow_pattern` label | ✔ 5 levels | ✘ absent |
| `direction_slow_aware` | ✔ | ✘ absent |

**Why it matters:** 5-HT2 receptor–mediated effects can develop slowly
(>10 s latency). Without the early/late split, true "slow biphasic" cells
(e.g. early inhibition → late excitation via Gq cascade) are called
mono-phasic excitation or no_change, depending on where the peak lands.

**Fix:** Add `early_window` and `late_window` parameters to
`extract_cell_features_one()`.  Compute per-window persistence flags and
extrema, then emit `slow_pattern` and `direction_slow_aware` columns.

### 2.2 Puff-window extrema and return-to-baseline latency

| Feature | `df_prep_new.R` | Promoted |
|---|---|---|
| `puff_window = c(0, 6)` | ✔ | ✘ |
| `puff_exc_mag` / `puff_inh_mag` | ✔ | ✘ |
| `time_to_return_baseline()` | ✔ full function | ✘ absent |
| `return_latency_s` | ✔ | ✘ |
| `peak_time_for_return` | ✔ | ✘ |

**Why it matters:**
- The puff window (0–6 s) captures the **immediate** transient response
  that is the signature of ionotropic/fast G-protein effects.  Without it,
  a cell that has a brief 2-s excitatory burst followed by 10-s inhibition
  can be misclassified because the rapid window (0–10 s) lumps both phases
  together.
- `return_latency_s` is used downstream by `flag_low_change()` to
  identify cells with tiny, transient excursions that should really be called
  `no_change`.

**Fix:** Restore `time_to_return_baseline()` in `classify5HT_helpers.R`.
Restore `puff_window` extrema computation inside `extract_cell_features_one()`.
Add the `return_latency_s` column to the output tibble.

### 2.3 `flag_low_change()` — the no-change safety net

The older pipe runs `flag_low_change()` **after** `add_biphasic_features()`:

```r
per_cell2 <- per_cell %>%
  add_biphasic_features() %>%
  flag_low_change(puff_mag_thr = 10, max_return_latency = 20) %>%
  dplyr::mutate(
    response_type = if_else(
      response_type %in% c("excitation","inhibition") & low_change_fast_return,
      "no_change", as.character(response_type))
  )
```

This reclassifies cells whose excursion is small **and** returns to baseline
quickly as `no_change`.  The promoted code has no equivalent safeguard, so
noisy cells with brief ±5-8 % blips get labelled as mild_excitation or
mild_inhibition when a human reviewer would call them flat.

**Fix:** Promote `flag_low_change()`.  Wire it into `build_per_cell_classifier()`
(or expose it as a standalone post-processing step that users call before
plotting).

### 2.4 Post-window duration and late-slope windows

| Parameter | `df_prep_new.R` | Promoted |
|---|---|---|
| `post_window` | `c(0, 56)` | `c(0, 60)` |
| `late_slope_window` | `c(36, 56)` | none (uses single `late_window`) |
| `late_mean_window` | `c(46, 56)` | none |

The promoted code collapses "late slope" and "late mean" into one
`late_window = c(40, 60)`, losing the ability to separately characterise
the **trend** in the last 20 s versus the **level** in the final 10 s.

**Fix:** Split `late_window` into `late_slope_window` and
`late_mean_window` parameters.  Update `params5HT$classifier` defaults:

```r
$classifier$late_slope_window  # c(36, 56)
$classifier$late_mean_window   # c(46, 56)
$classifier$post_window        # c(0, 56)
```

### 2.5 Smoother defaults

The promoted code defaults to `smooth_method = "rollmean", smooth_k = 3`.
The most mature dev version uses `smooth_method = "rollmean", k_smooth = 3`
in the **build script** but `rollmedian, k = 11` in the function's formal
defaults.

`rollmedian` is more robust to firing rate outliers (single-spike bursts that
can produce pct_of_baseline > 300 for one timepoint).  In instantaneous
firing-rate data, this matters.

**Recommendation:** Change the formal default in `extract_cell_features_one()`
to `smooth_method = "rollmedian"` while keeping `smooth_k = 3` (or allow
`params5HT` to set this).  The user can always override.

### 2.6 Manual overrides

`df_prep_new.R` contains hard-coded manual overrides for specific cells:

```
Q21.26.017.1A.18.02  → forced to inhibition (not biphasic)
QN25.26.014.11.12.01 → forced to biphasic inh_then_exc
QN26.26.004.20.01.01 → forced to biphasic inh_then_exc
```

These are embedded in runnable-script blocks and would be lost if anyone
re-runs the pipeline from scratch after ignoring the dev files.

**Recommendation:** Create a dedicated `manual_overrides_5HT` list or CSV
in `data-raw/` that the pipeline consumes.  `params5HT$overrides` could
point to the path or hold the data inline:

```r
params5HT$overrides <- list(
  force_inhibition = c("Q21.26.017.1A.18.02"),
  force_biphasic_inh_then_exc = c("QN25.26.014.11.12.01", "QN26.26.004.20.01.01")
)
```

Then `build_per_cell_classifier()` applies them at the very end.  This makes
overrides auditable and version-controlled instead of scattered inside ad-hoc
scripts.

## 3. Magnitude / response-call binning

The promoted `classify_magnitude()` uses hard-coded cut-offs:

```r
classify_magnitude <- function(mag, mild = 10, moderate = 25, strong = 50)
```

These thresholds are reasonable but should be surfaced through
`params5HT$classifier` so they can be tuned without editing source:

```r
params5HT$classifier$mag_mild     # 10
params5HT$classifier$mag_moderate # 25
params5HT$classifier$mag_strong   # 50
```

Also, **biphasic cells currently fall through to `"no_change"` in the
response_call logic** inside `extract_cell_features_one()`:

```r
response_call <- case_when(
  response_type == "excitation" & mag_bin == "mild" ~ "mild_excitation",
  ...
  TRUE ~ "no_change"    # <-- biphasic lands here!
)
```

Biphasic cells need their own label(s).  The simplest fix is to assign
`response_call = "biphasic"` with a new colour entry:

```r
response_cols["biphasic"] <- "#9970AB"   # purple
pretty_labels["biphasic"] <- "Biphasic"
```

Or you could go further and have `biphasic_exc_first` / `biphasic_inh_first`.

## 4. Plotting — what's broken, what's missing

### 4.1 Colour-scheme confusion

Three different colour schemes coexist across versions:

| File | Excitation colour | Inhibition colour |
|---|---|---|
| `various_visual_tools.R` | Blue (#2166AC) | Red (#B2182B) |
| `donut_plots.R` | Red (#FC9272–#CB181D) | Blue (#08306B–#6BAED6) |
| `df_prep_and_per_cell_build.R` | Red (#d62728) | Blue (#1f77b4) |
| Promoted `classify5HT_helpers.R` | Blue (#2166AC) | Red (#B2182B) |

There is no consensus.  The promoted palette follows `various_visual_tools.R`
(blue = excitation, red = inhibition), but other scripts do the opposite.

**Fix:** Pick one canonical mapping and document it in `params5HT`.  The
neuroscience convention (matching patch-clamp colour codes) is:

- **Blue** → inhibition / hyperpolarisation
- **Red** → excitation / depolarisation

If you prefer the inverted scheme, that's fine — just enforce it everywhere.
Add to `params5HT$colour`:

```r
params5HT$colour <- list(
  excitation = "#B2182B",
  inhibition = "#2166AC",
  biphasic   = "#9970AB",
  no_change  = "grey80"
)
```

Then the plotting functions should read from `params5HT$colour` so there is
exactly one place to change it.

### 4.2 Per-cell QC plot — missing peak markers and window labels

The promoted `plot_cell_qc()` draws shaded time windows and the smoothed trace
but does **not** overlay:

1. **Excursion peak markers** — coloured dots at the detected min/max times.
   (The dev version `plot_bucket_qc()` shows these as red/blue dots.)
2. **Late-fit line** — the linear regression through the late window that
   visualises whether the trace is recovering or drifting.
3. **Threshold lines** — the promoted version draws these, but the colour
   assignment is swapped relative to the newer dev scripts (blue for exc
   threshold, red for inh threshold — should match your chosen colour
   scheme).
4. **Subtitle diagnostics** — the dev version's subtitle shows baseline Hz,
   rapid pattern, abs_min/max, etc.  The promoted version only shows
   `assigned_subclass`.

**Fix:** Refactor `plot_cell_qc()` to accept an optional `per_cell` row so
it can overlay markers, late-fit line, and a fuller subtitle.  Keep the
current minimal signature as the default path for quick browsing.

### 4.3 Spline / mean-trace overlay

The promoted `plot_exc_inh_traces()` computes a per-group **arithmetic mean**
of the smoothed trace and draws it as a single black line.  Issues:

1. **No spline smoothing on the mean line.** The dev version
   (`make_mean_df()`) supports `mean_smooth = c("spline","rollmean","none")`
   with `smooth.spline(df = 20)`.  The raw mean can be spiky when cell counts
   vary across time bins; a spline fit removes that noise.

2. **Time-bin alignment.** The promoted code groups by exact `time` values,
   which only works if all cells share identical sample times.  If time grids
   are even slightly misaligned (different sampling rates or offsets), the
   "mean" is computed from very few cells at some timepoints, producing
   artefactual spikes.  The dev version (`make_plot_df_ei()`) bins time:
   ```r
   time_bin = round(time / time_bin_s) * time_bin_s
   ```
   and filters bins with `n_cells >= min_cells_per_time`.

3. **No N annotation.** The dev version carries `n_cells` per time bin so you
   can debug thin spots.

**Fix:**
- Add `time_bin_s` and `min_cells_per_time` parameters to
  `plot_exc_inh_traces()`.
- Apply time binning before computing the mean.
- Add optional spline-smooth pass on the group mean (controlled by a
  `mean_smooth` parameter).
- Optionally annotate N per bin as a secondary axis or ribbon.

### 4.4 Missing subtype faceting

The promoted `plot_exc_inh_traces()` facets by
`response_type ~ assigned_subclass` — that's correct for a coarse view.
But **biphasic cells are excluded** because the filter is:

```r
filter(response_type %in% c("excitation", "inhibition"))
```

The dev versions provide dedicated faceted views for:

- **`plot_biphasic_stacked_by_subclass()`** — stacked traces with
  vertical offsets, faceted by subclass, separated by biphasic order.
- **`plot_patchwork_inh_then_exc()` / `plot_biphasic_excfirst_patchwork()`**
  — multi-panel patchwork grids of individual biphasic cell traces, using
  `patchwork::wrap_plots()`.
- **`plot_stacked_inh_first_by_subclass()`** — proportion bar chart of
  biphasic orders by subclass.

None of these were promoted.

**Fix (gradual approach):**
1. Promote `plot_exc_inh_traces()` to handle `response_type = "biphasic"` by
   splitting biphasic cells by `biphasic_order` and using a 2×3 or 3×3 facet:
   `biphasic_order ~ assigned_subclass`.
2. Add a `plot_biphasic_traces()` function that wraps the stacked/patchwork
   approach from `dev/classify_pct_change/building_stacked_by_subclass.R`.
3. Add `plot_biphasic_composition()` to show biphasic-order proportions.

### 4.5 Split violin / centered violin

The promoted `plot_centered_violin()` plots `response_peak` (which is a
magnitude, not a raw % of baseline) and doesn't split by direction.  The dev
versions (`split_violins.R`, `split_violins_with_summarized_responses.R`)
introduce:

- True baseline-centred delta (`response_peak - 100`).
- A "gap" between the two sides so excitation and inhibition violins don't
  overlap.
- Axis labels that show the true delta, corrected for the gap.
- `coord_flip()` for horizontal display.

**Fix:** Replace `plot_centered_violin()` with a `plot_split_violin()` that
uses the gap-centred delta approach and colours each side by direction.

### 4.6 Donut / pie charts

**Not promoted.** These are in `donut_plots.R` and `various_visual_tools.R`.
They are presentation-quality figures that several dev scripts reference.

**Recommendation:** Optionally promote `plot_donuts_by_subclass()` into
`plot_classify5HT.R`.  Low priority compared to items above, but useful for
talk slides.

### 4.7 Excitation-magnitude bar charts (blockers analysis)

`summary_visuals.R` contains bar charts of excitation magnitude grouped by
`blockers_bin` (has_blockers vs no_blockers) with error bars and N labels.
The promoted code has no blocker-aware plotting at all.

**Recommendation:** Add `blockers` metadata to the classifier pipeline output
and create `plot_blocker_comparison()`.

## 5. Architectural improvements

### 5.1 Wire classifier defaults through `params5HT`

The promoted `classify_puff_response()` accepts 15 parameters with hard-coded
defaults.  These should read from `params5HT$classifier` so that tuning
happens in one place:

```r
classify_puff_response <- function(df, params = params5HT, ...) {
  p <- modifyList(params$classifier, list(...))
  df_prepped <- prep_puff_df(df,
    baseline_window = p$baseline_window,
    subclass_keep   = p$subclass_keep, ...)
  per_cell <- build_per_cell_classifier(df_prepped,
    rapid_window = p$rapid_window,
    min_excursion_pct = p$min_excursion_pct, ...)
  list(df_prepped = df_prepped, per_cell = per_cell)
}
```

### 5.2 Separate classification from feature extraction

Currently `extract_cell_features_one()` both extracts features **and** assigns
`direction`, `response_type`, and `response_call`.  This couples feature
extraction to classification logic, making it hard to experiment with
different decision rules without re-running smoothing.

**Recommended refactor:**

```
extract_cell_features_one()  → pure feature tibble (extrema, persistence,
                                slopes, latencies)
classify_cell_response()     → take feature row → add direction, response_type,
                                response_call, biphasic_order
apply_manual_overrides()     → take per_cell tibble → apply overrides from
                                params5HT$overrides
```

### 5.3 Make `add_biphasic_features()` unnecessary

The promoted code builds `response_type` and `biphasic_order` inside
`extract_cell_features_one()`.  The dev pipe has a separate
`add_biphasic_features()` step.  Both approaches exist because the extractor
in `df_prep_new.R` does **not** compute `response_type`.

In the unified refactor (5.2), the classify step should handle both
rapid-biphasic and slow-biphasic detection in one pass, eliminating the need
for a separate post-hoc `add_biphasic_features()`.

### 5.4 Consistent column-naming convention

Different versions use different column names:

| Concept | `df_prep_new.R` | `df_prep_and_per_cell_build.R` | Promoted |
|---|---|---|---|
| Smooth trace | `pct_smooth` | `pct_smooth` | stored in local `y_smooth` only |
| Baseline Hz | `baseline_hz` | `baseline_hz` | `baseline_hz` ✔ |
| Rapid persistence | `rapid_has_exc_persist` | `rapid_has_exc_persist` | `rapid_has_exc` |
| Biphasic flag | `is_biphasic_any` | `is_biphasic_rapid` | (inferred from response_type) |

**Fix:** Adopt the `_persist` suffix consistently, since the flag **is**
persistence-gated.  Document the column dictionary in a roxygen block for
`build_per_cell_classifier()` or in a separate vignette.

## 8. Conclusion

The classifier's core architecture (baseline normalisation → smooth → persistence-gated
feature extraction → rule-based classification) is sound.  The main problems are:

1. **Feature regression** — the promoted code dropped the early/late window split,
   puff-window analysis, return-to-baseline latency, and no-change safety net that
   the most mature dev version had.
2. **Plotting regression** — spline overlay, peak markers, subtype faceting, and
   consistent colouring were lost.
3. **Biphasic gap** — biphasic cells fall through to `no_change` in the response_call
   mapping.
4. **Configuration sprawl** — hard-coded defaults instead of `params5HT` lookup.

Fixing items P0/P1 from section 6 would restore the classifier to the capability
level of the best dev version while keeping the clean R-package structure.
